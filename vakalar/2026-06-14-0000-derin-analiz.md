## 🔟 EN İLGİNÇ 10 PROJE

---

### 1. Hyper (YC P26) — "Company Brain to Power Agentic Development"
- **Kim:** Y Combinator S26 batch, Hyper HQ
- **Ne:** Şirketlerin tüm dokümanlarını, kodlarını, Slack mesajlarını, Jira ticketlerını tek bir "company brain"da birleştiren AI agent platformu. Kod yazarken şirketin apisini, standartlarını, geçmiş kararlarını otomatik hatırlayan bir记忆力 sistemi.
- **Tech:** Python, Claude/GPT modelleri, embedding tabanlı retrieval
- **Sayı:** ▲79 💬76 (HN Launch)
- **Neden çıkıyor:** Ekiplerde AI coding agent'ların "context" sıkıntısını çözüyor — agent sadece mevcut dosyayı değil, şirketin tüm bilgi tabanını biliyor.
- **Herkesin kaçırdığı nokta:** Hyper'ın asıl sattığı şey "context window" değil, "kurumsal hafıza". Her şirketin kendine özgü apis'i, naming convention'ları, tech debt'i var — agent bunları bilmeden kod yazınca üç kez revize gerekiyor. Hyper bunu çözüyor.

---

### 2. Intuned (YC S22) — "Build and Run Reliable Browser Automations as Code"
- **Kim:** Y Combinator S22 batch, Intuned HQ
- **Ne:** Browser automation'ları "code as" olarak yazıp çalıştıran platform. Selenium/Playwright yerine declarative tanımlar, agent'lar browser'da görev yapabiliyor.
- **Tech:** Node.js, browser automation, YC backing
- **Sayı:** ▲117 💬57 (HN Launch — en yüksek score'lu AI agent launch'larından biri)
- **Neden çıkıyor:** Herkes "agent browser" diyor ama kimse güvenilir şekilde production'a taşıyamıyordu. Intuned bunu "reliable" yapan — agent browser'da tıkla, form doldur, veri çek ama artık brittle değil.
- **Herkesin kaçırdığı nokta:** İntuned'ın arkasındaki fikir: "Browser automation = yeni API." Geleneksel API'ler yetişmediği yerde (LinkedIn scraping, dynamic web forms, multi-step workflows) browser tabanlı agent'lar yeni standart. Ama kimse bunu bu kadar düzgün package etmemişti.

### 3. Agora-1 — "The Multi-Agent World Model"
- **Kim:** Odyssey (odyssey.ml), bağımsız araştırma ekibi
- **Ne:** Çoklu agent'ların birlikte çalışıp gerçek dünya görevlerini yerine getirdiği bir "world model". Agent'lar birbirleriyle koordine olup incomplete information altında karar veriyor.
- **Tech:** LLM multi-agent coordination, world model framework
- **Sayı:** ▲128 💬23 (en yüksek score'lu projelerden)
- **Neden çıkıyor:** Tek agent'lar güçlü ama çok agent'lar = yeni bir zeka katmanı. Agora bunu sistematik hale getiriyor.
- **Herkesin kaçırdığı nokta:** Çoğu insan "multi-agent" deyince "birden fazla coding agent" düşünüyor. Ama Agora-1'in yaptığı farklı: gerçek dünya görevlerinde (ulaşım, lojistik, koordinasyon) agent'ların birbirine bağımlı kararlar vermesi. Bu = gelecekteki "agent economy"nin temeli.

### 4. Runtime (YC P26) — "Sandboxed Coding Agents for Everyone on a Team"
- **Kim:** Y Combinator P26 batch, Runtime
- **Ne:** Ekip içinde coding agent'ları sandboxed ortamda çalıştıran platform. Her developer kendi agent'ını kurabiliyor, ama şirket verilerine erişim kontrollü.
- **Tech:** YC backing, sandbox isolation, team collaboration
- **Sayı:** ▲103 💬30
- **Neden çıkıyor:** Şirketler AI agent'ları ekibe sokmak istiyor ama güvenlik/izolasyon sorunu var. Runtime bunu "team-ready" hale getiriyor.
- **Herkesin kaçırdığı nokta:** Security değil, **ownership** sorununu çözüyor. Agent kimın verisiyle ne yapabilir? Bugün çoğu şirket bunu yönetemiyor. Runtime "her developer + kendi agent'ı + kontrollü erişim" modelini sunuyor.

### 5. Open Envelope — "Open Schema for Defining AI Agent Teams"
- **Kim:** ashconway (bağımsız geliştirici)
- **Ne:** AI agent'ların takım olarak nasıl çalışacağını tanımlayan açık standart/schema. "Agent team manifestosu" — kim ne yapar, nasıl iletişim kurar, hata durumunda ne olur.
- **Tech:** JSON/Mermaid tabanlı schema, açık standart
- **Sayı:** ▲52 💬13
- **Neden çıkıyor:** Agent'lar arası iletişim = kaotik. Her ekip kendi iletişim protokolünü yazıyor. Open Envelope bunu standartlaştırıyor.
- **Herkesin kaçırdığı nokta:** Open Envelope aslında bir "agentic git" — kod için git nasılsa, agent iletişimi için de bu schema gerekli. gelecekte tüm agent'lar bu standartı kullanacak.

### 6. InsForge — "Open-source Heroku for Coding Agents"
- **Kim:** InsForge ekibi (github.com/InsForge/InsForge)
- **Ne:** Coding agent'lar için self-hosted deployment platformu. Agent'ı kur, deploy et, API key ver, çalışsın. Heroku gibi ama sadece agent'lar için.
- **Tech:** Python, Docker, self-hosted, açık kaynak
- **Sayı:** ⭐62 (GitHub stars, HN'de de ilgi görmüş)
- **Neden çıkıyor:** Agent'ları bulutta çalıştırmak pahalı. InsForge self-hosted seçeneği sunuyor — kendi sunucunda, kendi API key'inle.
- **Herkesin kaçırdığı nokta:** AI coding agent'lar için "hosting" = yeni Heroku fırsatı. Bugün herkes API key'le SaaS kullanıyor ama 6 ay sonra her şirket kendi agent fleet'ini host etmek isteyecek. InsForge bunu açık kaynak yapıyor.

### 7. "I Nerfed Our Coding Agents on Purpose" — noahfradin
- **Kim:** noahfradin (bağımsız geliştirici)
- **Ne:** Coding agent'ların yeteneklerini kasıtlı olarak kısıtlayarak (nerf) daha kontrollü, daha az hata yapan sistemler kurma deneyimi. Agent'ın "süper güçlerini" bilinçli azaltmak.
- **Tech:** Claude Code, agent configuration, rate limiting
- **Sayı:** ▲27 💬10
- **Neden çıkıyor:** Herkes agent'ları daha güçlü yapmaya çalışıyor. Ama bazen daha az güç = daha az hata = daha az API maliyeti. Paradoks gibi ama çalışıyor.
- **Herkesin kaçırdığı nokta:** "Agent'ı güçlendirmek" yerine "agent'ı akıllıca kısıtlamak" = yeni bir disiplin. noahfradin'in bulgusu: agent'ın MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES=3 kuralı gibi küçük constraint'ler büyük fark yaratıyor. Bunu yapan yok.

### 8. Kanban CLI — "Local-first, Agent-first Task Manager for Terminal"
- **Kim:** hydrafog (codeberg.org)
- **Ne:** Terminalde çalışan, agent-first task manager. Agent'lar terminal üzerinden görev yönetimi yapabiliyor. Git tabanlı, local-first.
- **Tech:** Rust (muhtemelen), terminal UI, git-based storage
- **Sayı:** ▲17 💬13
- **Neden çıkıyor:** Agent'lar için "task management" = yeni ihtiyaç. Agent ne yapacağını bilmeli, sıralamalı, takip etmeli.
- **Herkesin kaçırdığı nokta:** Çoğu insan "agent memory"den bahsediyor ama "agent task manager" = çok daha pratik. Agent'ın yapılacaklar listesi, öncelikleri, tamamlanma durumu — bunu terminalde yönetmek = yeni bir UX pattern.

### 9. LangMCP — "AI Agents Need Inspectable State"
- **Kim:** Medium makale, bağımsız geliştirici
- **Ne:** AI agent'ların internal state'ini (hafıza, karar süreçleri, ara sonuçlar) inspect edilebilir, debug edilebilir hale getiren MCP (Model Context Protocol) extension.
- **Tech:** MCP, debugging, state inspection
- **Sayı:** ▲2 💬2 (küçük ama önemli konsept)
- **Neden çıkıyor:** Agent'lar karar verdiklerinde "nasıl karar verdin" sorusu cevapsız kalıyor. LangMCP bunu açık hale getiriyor.
- **Herkesin kaçırdığı nokta:** Agent debugging = gelecekteki en büyük engineering challenge. Bugün "agent neden yanlış yaptı" sorusu cevapsız kalıyor. LangMCP gibi tool'lar = agent observability'nin temeli.

## ✅ 5 TAVSİYE

---

### 1. Hyper — "Company Brain" Modelini Kendi İşine Uygula
**Neden popüler:** Çünkü AI coding agent'ların en büyük sıkıntısı "context eksikliği". Agent şirketin naming convention'ını, API standartlarını, daha önce yapılan hataları bilmeden kod yazıyor ve sürekli düzeltme gerekiyor.

**Kime lazım:** Software şirketleri, özellikle 10+ developer olanlar. Her developer farklı AI agent kullanıyor ama agent'lar şirketin "kurumsal hafızasından" yoksun.

**Nasıl yapılır:**
1. Şirketin tüm dokümanlarını (README, API docs, architecture decision records) bir embedding database'e koy
2. Her agent request'inde "company context" injection yap
3. Retrieval-augmented generation (RAG) ile agent'a şirket bilgisini ver
4. Yeni dokümanlar eklendikçe embedding'i güncelle

**İhtiyaçlar:** Embedding modeli (OpenAI ada-002 veya benzeri), vector DB (Pinecone, Weaviate, veya local Qdrant), RAG pipeline

---

### 2. Intuned Browser Automation Modeli — "Yeni API" Fırsatı
**Neden popüler:** Geleneksel API'ler dinamik web sitelerinde çalışmıyor. LinkedIn, Instagram, kompleks SaaS dashboard'ları = API yok, sadece browser var. Intuned bunu "code as" hale getirerek yeni bir pattern yarattı.

**Kime lazım:** E-ticaret operasyonları, sosyal medya yönetimi, web scraping ihtiyacı olan herkes.

**Nasıl yapılır:**
1. Playwright veya Puppeteer ile browser automation script'leri yaz
2. Her adımı "step-by-step" declarative olarak tanımla (tıkla → bekle → doldur → çek)
3. Agent'a browser kontrolü ver — sadece API değil, gerçek browser
4. Sonuçları structured data olarak çıkar

**İhtiyaçlar:** Playwright/Puppeteer, Chrome extension veya headless browser, structured output parser

---

### 3. Open Envelope Standartı — Agent Takım Manifestosu Yaz
**Neden popüler:** Agent'lar arası iletişim standartsız. Her ekip kendi "agent protocol"ünü yazıyor. Open Envelope gibi açık standart = gelecekteki "agent internet"in TCP/IP'si olabilir.

**Kime lazım:** Çoklu agent sistemleri kuranlar, agent workflow'ları yönetenler.

**Nasıl yapılır:**
1. Agent takımındaki her agent'ın rolünü tanımla (orchestrator, worker, reviewer)
2. Agent'lar arası mesaj formatını JSON schema ile standartlaştır
3. Hata durumunda fallback rolleri belirle
4. Open Envelope schema formatını kullan (veya kendin benzer bir şey tasarla)

**İhtiyaçlar:** JSON schema bilgisi, agent orchestration framework (LangGraph, Mastra, veya custom)

---

### 4. "Nerf" Tekniği — Agent'ı Bilinçli Kısıtla
**Neden popüler:** Agent'ların "süper güç" modunda çalışması = yüksek hata oranı + yüksek API maliyeti. noahfradin'in gösterdiği gibi: kasıtlı kısıtlamalar (rate limit, max retries, bounded context) hem daha güvenilir hem daha ucuz sistemler üretiyor.

**Kime lazım:** Production'da AI agent kullanan herkes, özellikle maliyet endişesi olanlar.

**Nasıl yapılır:**
1. Agent config'inde MAX_* limitleri belirle (MAX_CONSECUTIVE_FAILURES, MAX_API_CALLS_PER_TASK, vb.)
2. "Circuit breaker" pattern'i uygula — 3 başarısızlıktan sonra dur ve raporla
3. Agent'ı sadece 1-2 adım atlat, her adımda checkpoint al
4. Maliyet budget'i koy — agent'ın harcadığı token'ı izle

**İhtiyaçlar:** Agent config bilgisi, logging/monitoring (LangSmith, Weights & Biases veya basit Prometheus)

---

### 5. SeaTicket Multi-Channel Modeli — Müşteri Destek Agent'ı
**Neden popüler:** Müşteri desteği = en pahalı insan iş gücü. Agent GitHub issues, email ve forum'ları tek bir görünümde birleştirip otomatik yanıt veriyor. Multi-channel = müşteri deneyimi = sadakat.

**Kime lazım:** SaaS şirketleri, e-ticaret siteleri, her gün 50+ destek talebi alanlar.

**Nasıl yapılır:**
1. GitHub API, email (IMAP/SMTP), forum API'lerini bağla
2. Her kaynaktan gelen mesajları normalize et (farklı formatları tek modele çevir)
3. Aynı müşterinin farklı kanallardaki taleplerini bağla (email adresi + GitHub username eşleştirmesi)
4. Agent'a "triage" yetkisi ver — hangisi otomatik çözülür, hangisi insana yönlendirilir

**İhtiyaçlar:** Multi-source API entegrasyonu, entity resolution (aynı kişiyi farklı kanallarda tanıma), confidence thresholding
